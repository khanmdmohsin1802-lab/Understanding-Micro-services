## 🐳 Root File: `docker-compose.yml`

**Purpose:** Defines and runs both database containers using Docker so you don't need to manually install PostgreSQL or MongoDB locally.

```yaml
version: '3.8'
services:
  postgres-user-db:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: admin
      POSTGRES_PASSWORD: secretpassword
      POSTGRES_DB: user_service_db
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data

  mongodb-catalog:
    image: mongo:6.0
    environment:
      MONGO_INITDB_ROOT_USERNAME: admin
      MONGO_INITDB_ROOT_PASSWORD: secretpassword
    ports:
      - "27018:27017"
    volumes:
      - mongodata_v3:/data/db

volumes:
  pgdata:
  mongodata_v2:
  mongodata_v3:
```

**What each block does:**
- `postgres-user-db` → Runs a PostgreSQL 15 database container. The DB is named `user_service_db`. It maps port `5432` on your machine to the container. Data persists in the `pgdata` Docker volume so it's not lost when the container restarts.
- `mongodb-catalog` → Runs a MongoDB 6.0 container. It maps your machine's port `27018` to the container's `27017`. Data persists in `mongodata_v3`.
- `volumes` block → Declares the named volumes so Docker knows to persist them.

> **Note:** `mongodata_v2` is declared but unused — likely a leftover from a volume rename.

## 📄 `controllers/authController.js` — Register & Login Logic

```js
import bcrypt from "bcrypt";
import jwt from "jsonwebtoken";
import db from "../config/db.js";

// ── REGISTER ──────────────────────────────────────────────
const registerUser = async (req, res) => {
  const { email, password, role } = req.body;

  try {
    const userExists = await db.query("SELECT id FROM users WHERE email = $1", [email]);
    if (userExists.rows.length > 0) {
      return res.status(409).json({ error: "User Already Exists" });
    }

    const salt = await bcrypt.genSalt(10);
    const passwordHash = await bcrypt.hash(password, salt);

    const newUser = await db.query(
      "INSERT INTO users (email, password_hash, role) VALUES ($1, $2, $3) RETURNING id, email, role",
      [email, passwordHash, role || "customer"]
    );

    res.status(201).json({
      message: "User registerd successfully",
      user: newUser.rows[0],
    });
  } catch (error) {
    console.log(error.message);
    res.status(500).json({ error: "Internal server error" });
  }
};

// ── LOGIN ─────────────────────────────────────────────────
const loginUser = async (req, res) => {
  const { email, password } = req.body;

  try {
    const result = await db.query("SELECT * FROM users WHERE email = $1", [email]);

    if (result.rows === 0) {
      return res.status(401).json({ message: "Invalid Credentials" });
    }

    const user = result.rows[0];

    const isMatch = await bcrypt.compare(password, user.password_hash);
    if (!isMatch) {
      return res.status(401).json({ message: "Invalid Credentials" });
    }

    const token = jwt.sign(
      { id: user.id, role: user.role },
      process.env.JWT_SECRET,
      { expiresIn: "24h" }
    );

    res.status(200).json({
      message: "Login Successful",
      token: token,
      user: { id: user.id, email: user.email, role: user.role },
    });
  } catch (error) {
    console.error("Login error", error.message);
    res.status(500).json({ error: "Internal Server Error" });
  }
};

export { registerUser, loginUser };
```

### `registerUser` — What it does step by step:
1. Extracts `email`, `password`, `role` from the request body
2. Queries the DB to check if a user with that email already exists → returns `409 Conflict` if yes
3. Generates a bcrypt salt (complexity factor 10) and hashes the password — **never stores plain text passwords**
4. Inserts the new user into the `users` table with `email`, `password_hash`, and `role` (defaults to `"customer"` if not provided)
5. Returns the newly created user (id, email, role) with `201 Created`

### `loginUser` — What it does step by step:
1. Extracts `email` and `password` from the request body
2. Fetches the user record from the DB by email
3. Uses `bcrypt.compare()` to check if the plain-text password matches the stored hash — returns `401` if not
4. Signs a **JWT token** containing `{ id, role }` with a 24-hour expiry
5. Returns the token + user info on success

> ⚠️ **Known Bug:** `result.rows === 0` on line 46 should be `result.rows.length === 0` — the current check will never trigger because `rows` is an array, not a number.

## 📄 `middlewares/requireAuth.js` — JWT Guard

```js
import jwt from "jsonwebtoken";

const requireAuth = (req, res, next) => {
  const authHeader = req.headers.authorization;

  if (!authHeader || !authHeader.startsWith("Bearer ")) {
    return res.status(401).json({
      message: "Unauthorized: Missing or Invalid token",
    });
  }

  const token = authHeader.split(" ")[1];

  try {
    const decode = jwt.verify(token, process.env.JWT_SECRET);
    console.log(decode);
    req.user = decode;
    next();
  } catch (error) {
    res.status(403).json({ error: "Forbiden: Invalid or Expired token" });
  }
};

export { requireAuth };
```

**What it does:**
- Reads the `Authorization` header from the incoming request
- Validates it starts with `"Bearer "` — rejects with `401` if missing or malformed
- Splits the header to extract just the token string
- Calls `jwt.verify()` with the secret key — if valid, decodes the payload
- Attaches the decoded payload (`{ id, role }`) to `req.user` so downstream controllers can access it
- Calls `next()` to continue to the controller; if token is invalid/expired → `403 Forbidden`

## 📄 `middlewares/validateRequest.js` — Schema Validation Factory

```js
const validateRequest = (schema) => {
  return (req, res, next) => {
    try {
      schema.parse(req.body);
      next();
    } catch (error) {
      return res.status(400).json({
        error: "validation fail",
        details: error.errors,
      });
    }
  };
};

export { validateRequest };
```

**What it does:**
- A **higher-order function** (a function that returns another function) — this pattern is called a middleware factory
- Takes a Zod `schema` as argument
- Returns an Express middleware function that calls `schema.parse(req.body)`
- If validation passes → calls `next()` to proceed to the controller
- If validation fails → immediately returns `400 Bad Request` with detailed Zod error messages
- This allows you to reuse the same validation pattern for any route just by passing a different schema

## 📄 `schemas/authSchema.js` — Zod Validation Schemas

```js
import { z } from "zod";

const registerSchema = z.object({
  email: z.string().email({ message: "Invalid Email" }),
  password: z
    .string()
    .min(6, { message: "Password must atleast 6 character long" })
    .regex(/[A-Z]/, { message: "Password must have atleaset one upper case letter" })
    .regex(/^[a-zA-Z0-9]/, { message: "Password must contain one speacial character" }),
  role: z.enum(["customer", "admin", "venue_manager"]).optional(),
});

const loginSchema = z.object({
  email: z.string().email({ message: "Invalid email address" }),
  password: z.string().min(1, { message: "Password is Required" }),
});

export { registerSchema, loginSchema };
```

**`registerSchema` rules:**
- `email` → Must be a valid email address format
- `password` → Must be at least 6 characters, must contain at least one uppercase letter, must start with alphanumeric (note: the regex `/^[a-zA-Z0-9]/` only checks the start, not that a special character exists — this is a logic bug)
- `role` → Optional field; if provided must be one of `"customer"`, `"admin"`, or `"venue_manager"`

**`loginSchema` rules:**
- `email` → Valid email format
- `password` → Any non-empty string (just checking it's present)

> ⚠️ **Known Bug:** The special character regex `/^[a-zA-Z0-9]/` only checks that the password *starts* with alphanumeric — it doesn't actually enforce that a special character exists. Should be something like `regex(/[^a-zA-Z0-9]/, ...)`.

```js
class AppError extends Error {

    super(message);
    this.statusCode = statusCode;
    Error.captureStackTrace(this, this.constructor);
  }
```

**What it does:**

## 📄 `utils/asyncHandler.js` — Async Error Wrapper

```js
const asyncHandler = (fn) => {
  return (req, res, next) => {
    Promise.resolve(fn(req, res, next)).catch(next);
  };
};

export { asyncHandler };
```

**What it does:**
- Another **higher-order function** (middleware factory pattern)
- Wraps any async route handler so you don't need to write `try/catch` in every controller
- If the async function throws an error, `Promise.resolve(...).catch(next)` catches it and passes the error to `next(err)`, which then triggers the global `errorHandler` middleware
- Without this, unhandled promise rejections in async Express handlers would crash the server or silently fail
