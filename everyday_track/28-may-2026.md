## 🐛 Bug Fixes

### Fix 1 — `authController.js` › `loginUser`
**File:** `user-service/controllers/authController.js`

**The bug:**
When checking if a user exists in the database, the condition was comparing an array to a number — which can never be true.

```js
// ❌ Before (broken — rows is an array, not a number)
if (result.rows === 0) { ... }

// ✅ After (correct — check the length of the array)
if (result.rows.length === 0) { ... }
```

**Why it matters:**
Without this fix, the "user not found" check never triggered. So if someone tried to log in with an email that doesn't exist in the DB, the code would crash trying to access `result.rows[0]` (which would be `undefined`) instead of returning a clean `401 Invalid Credentials`.

### Fix 2 — `authSchema.js` › `registerSchema`
**File:** `user-service/schemas/authSchema.js`

**The bug:**
The special character validation regex was wrong — it was checking that the password *starts with* an alphanumeric character, not that it *contains* a special character.

```js
// ❌ Before (wrong — only checks the START of the string)
.regex(/^[a-zA-Z0-9]/, { message: "Password must contain one special character" })

// ✅ After (correct — checks anywhere in the string for a non-alphanumeric char)
.regex(/[^a-zA-Z0-9]/, { message: "Password must contain one special character" })
```

**Why it matters:**
The `^` inside a character class `[^...]` means "NOT these characters" — i.e., find any character that is NOT a letter or number. Without this fix, passwords like `Password1` (no special char) would pass validation when they shouldn't.

## ✅ New Files Added

### New File 1 — `validateRequest.js`
**Path:** `catalog-services/middlewares/validateRequest.js`
**Purpose:** A middleware factory that validates the entire request object (body, params, query) against a Zod schema.

**Pseudo-code:**
```
function validateRequest(schema):
  return async middleware(req, res, next):
    try:
      parsedData = await schema.parseAsync({
        body:   req.body,
        params: req.params,
        query:  req.query
      })

      // Replace request parts with sanitized/parsed values
      req.body   = parsedData.body
      req.params = parsedData.params
      req.query  = parsedData.query

      next()  // proceed to controller

    catch error:
      next(error)  // pass ZodError to global error handler
```

**Key concepts introduced:**

- **`schema.parseAsync()`** instead of `schema.parse()` — supports async Zod refinements (e.g., checking DB uniqueness in a schema)
- **Validates the whole request** (`body` + `params` + `query`) as one object, not just `req.body` — much more flexible
- **Reassigns parsed values back to req** — this means if Zod transforms or strips any fields, the controller receives the clean version automatically
- **Passes errors to `next(error)`** — lets the global `errorHandler` middleware handle ZodErrors uniformly instead of catching them here

**Difference from user-service's version:**
The user-service has a simpler version that only validates `req.body` synchronously. This catalog-service version is more advanced — it's async, validates the full request object, and sanitizes the data before the controller sees it.

### New File 2 — `catalogSchema.js`
**Path:** `catalog-services/schemas/catalogSchema.js`
**Purpose:** Zod schema for validating create-event requests in the catalog service.

**Pseudo-code:**
```
createEventSchema = z.object({
  body: z.object({
    title:       string, min 3 chars
    type:        enum ["movie", "concert", "comedy_show", "tech_conference"]
    description: string, min 10 chars, OPTIONAL
    metadata:    object, OPTIONAL
    showtimes:   non-empty array of {
      startTime:      ISO datetime string
      availableSeats: non-negative integer
      price:          non-negative number
    }
  })
})
```

**Field rules explained:**

| Field | Rule | Why |
|---|---|---|
| `title` | min 3 chars | Prevents empty or meaningless names |
| `type` | strict enum | Only 4 valid event types exist |
| `description` | min 10, optional | If provided, must be meaningful |
| `metadata` | any object, optional | Flexible extra data per event type |
| `showtimes` | non-empty array | An event with no showtimes is useless |
| `showtimes.startTime` | ISO datetime string | Zod validates proper date format |
| `showtimes.availableSeats` | non-negative int | Can't have -5 seats or 2.5 seats |
| `showtimes.price` | non-negative number | Can't have negative price |

**Why wrap inside `body: z.object({...})`?**
Because the new `validateRequest` middleware passes `{ body, params, query }` as one object to `parseAsync`. So the schema has to mirror that structure — the event fields live inside `body`.

## ✏️ Modified Files

### Modified 1 — `errorHandler.js`
**File:** `catalog-services/middlewares/errorHandler.js`
**Change:** Added a dedicated Zod error handler block at the top.

**Before (pseudo-code):**
```
errorHandler(err, req, res, next):
  respond with err.statusCode or 500
  respond with err.message or "Internal Server Error"
  in dev: include stack trace
```

**After (pseudo-code):**
```
errorHandler(err, req, res, next):

  // NEW: Handle Zod validation errors specially
  if err is a ZodError:
    return 400 {
      success: false,
      message: "Validation Error",
      errors: [
        { field: "body.title", message: "Title must be at least 3 characters long" },
        { field: "body.showtimes", message: "..." },
        ...
      ]
    }

  // Existing: Handle everything else
  respond with err.statusCode or 500
  respond with err.message or "Internal Server Error"
  in dev: include stack trace
```

**Why this matters:**
Before this change, if Zod threw a validation error it would fall into the generic 500 handler and the response would be confusing. Now ZodErrors get a clean `400 Bad Request` with a structured list of exactly which fields failed and why. The `field` value uses dot-notation like `body.showtimes.0.price` so the caller knows precisely what went wrong.
